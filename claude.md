3-3. UIポイント
入力欄：プレースホルダーにサンプルメモを薄く表示（入力したら消える）
整形欄：整形前から項目名＋✱マークを表示。初めてのユーザーが「こんな項目があるのね」と理解できる
✱マーク：必須項目のみ。薄い色で小さく表示（うるさくならないように）
モバイル：縦並びでも同じUXが成立する構成
ループ：①〜整形結果表示は何度でも繰り返せる。納得したらコピー or 部長へ

5. プロンプト設計
5-1. Groq①：整形処理
材料となる3つのJSONを組み合わせてプロンプトを自動生成する。

材料
内容
項目定義JSON
何の項目に入れるか・必須かどうか・選択肢
用語辞書JSON
業界用語の解釈（空配列からスタートし使いながら育てる）
ユーザーのメモ
実際の入力テキスト


プロンプトのポイント：
出力形式をJSONで明示（パース可能な構造で返ってくる）
「不明な場合は空文字」で補完・ハルシネーション防止
「説明文・前置き・```は不要」でパース失敗を防ぐ
選択肢がある項目（商談フェーズ・確度）は選択肢を列挙
確度のみ特別扱い：「メモに明記されている場合のみ入力。不明な場合は必ず空文字」

5-2. Groq②：複数案件判定（ヒソヒソくん用）
Yes/Noのみ返す。トークン節約・判定ブレ防止。

{"is_multiple": true または false}

判定基準：
- 会社名が複数登場する → true
- 「あと〜」「別件で〜」などの切り替えワードがある → true
- 上記に当てはまらない → false
- 迷ったら false（止めない・通す方針）

5-3. Groq③：プルプル部長
キャラクター設定をsystem_promptに埋め込む。時間・曜日コンテキストはPython側で判定してプロンプトに追加。

条件
部長の動作
平日18時未満
通常モード（武勇伝・ゴルフ・バブル時代・ケチ一言＋承認）
平日18時以降
コンプライアンス一言 → 昔話に脱線
休日
コンプライアンス一言 → 昔話に脱線


出力形式：
{"stamp": "承認：プルプル部長", "comment": "部長の一言"}

6. ヒソヒソくんの動作設計
判定方法
条件・発言
Python側
必須項目が空 →「…✱の{項目名}、聞けましたか？（小声）」
Groq②側
複数案件混在 →「たくさんご要望がありますね。。（小声）」
どちらでもない
何も言わない（沈黙）


任意項目が空でもヒソヒソくんは何も言わない（メリハリをつける）
必須項目が空のときだけ出てくるので、出現時の重みが増す
「止めない・通す」方針。気づかせるだけでユーザーに判断を委ねる

7. キャラクター設定
7-1. プルプル部長
58歳、昭和の営業マン気質
口癖：「わしの時代はな…」
メモに人名があれば必ず反応する（知ってるかどうかは関係ない）
ケチをつけるが最終的には必ず承認する
ケチのネタ：ひらがな・句読点・文字数・改行・カタカナ英語
雑談ネタ：ゴルフ・昔の接待文化・バブル時代・最近の若者への愚痴
18時以降・休日はコンプライアンスに配慮（ギャップ演出）

7-2. ヒソヒソくん（秘書秘書）
ヒソヒソ話すが内容は的確。実は一番仕事できる
話し方：「…あの、ちょっとよろしいですか（小声）」
入力する人の味方。指摘はするがうるさくしない
st.empty()で同じ場所を上書き表示（ズラズラ増えない）

8. 項目定義JSON（商談用・確定版）
{
  "template_name": "商談レコード",
  "fields": [
    { "name": "会社名",        "required": true },
    { "name": "担当者名",       "required": true },
    { "name": "商談フェーズ",   "required": true,
      "options": ["初回接触", "提案中", "見積提出", "クローズ"] },
    { "name": "予算感",         "required": false },
    { "name": "導入希望時期",   "required": false },
    { "name": "確度",           "required": false,
      "options": ["高", "中", "低"],
      "note": "メモに明記されている場合のみ入力。不明な場合は必ず空文字" },
    { "name": "競合情報",       "required": false },
    { "name": "次のアクション", "required": true },
    { "name": "備考",           "required": false }
  ]
}

備考フィールドの役割：
どの項目にも分類できなかった情報の受け皿
複数案件が混ざった場合も備考に入るので「変な報告書」が一目でわかる

9. 用語辞書JSON
{
  "glossary": []
}

仕組みだけ最初から作る（中身は空配列でスタート）
使いながら気づいた業界用語を追加していく運用
例：{ "term": "MM単価", "description": "月単価のこと", "field": "予算感" }

10. その他追加仕様
【全体イメージ】
- テーマカラーは、ブルーグレーや水色と、たまご色やオレンジの組み合わせ
- 薄いグレーの背景画像を用意する想定

【タイトルバー】
- 上部固定（スクロールで消えないこと）高さは30pxで、Streamlitのメニューバーは消すこと

【ハンコ仕様】
- 表示タイミング：部長コメント表示後、少し間を置いてドン
- ビジュアル：CSSで実装（画像不要）
  - 円形・朱色・かすれ風（opacity + filter: blur）
  - 桜の花びらを周囲に配置（🌸またはCSS擬似要素）
  - 文字：「承認」「プルプル部長」縦書き風
- アニメーション：scale 1.3 → 1.0 → 微振動で着地

【部長ページ レイアウト】
- st.columns([3, 1])で左右分割
  - 左：🐼 + 部長コメント（吹き出し風）
  - 右：ハンコ（コメント表示後にドン）
- その下：整形済み報告書パネル（session_stateから表示）
- 最下部：「戻る」ボタン

【コピーボタン】
- JavaScriptのクリップボードAPI方式
- ゆるゆるコックさんと同じ実装で

【キャラクタービジュアル】
- プルプル部長：🐼（パンダ）
- ヒソヒソくん：🐰（ウサギ）

- ヒソヒソくんのエラー演出は今回は不要
