```

「クリップボードにコピーしとく」ボタン  |  「プルプル部長に見せてみる🐼」ボタン
```

### 4-3. UIポイント
- テキストエリア：付箋スタイル（#FFFDE7・黄色ボーダー）。プレースホルダーにサンプルメモ表示
- 整形欄：整形前から項目名＋✱マーク表示。初回ユーザーが項目を理解できる
- ✱マーク：必須項目のみ。薄色・小サイズで表示（うるさくならないように）
- ループ：①〜整形結果表示は何度でも繰り返せる。納得したらコピー or 部長へ

## 6. プロンプト設計

### 6-1. Groq①：整形処理
材料となる3つのJSONを組み合わせてプロンプトを自動生成する。

| 材料 | 内容 |
|------|------|
| 項目定義JSON | 何の項目に入れるか・必須かどうか・選択肢 |
| 用語辞書JSON | 業界用語の解釈（空配列からスタートし使いながら育てる） |
| ユーザーのメモ | 実際の入力テキスト |

プロンプトのポイント：
- 出力形式をJSONで明示（パース可能な構造で返ってくる）
- 「不明な場合は空文字」で補完・ハルシネーション防止
- 「説明文・前置き・\`\`\`は不要」でパース失敗を防ぐ
- 選択肢がある項目（商談フェーズ・確度）は選択肢を列挙
- 確度のみ特別扱い：「メモに明記されている場合のみ入力。不明な場合は必ず空文字」
- `note` フィールドがある項目（導入希望時期・今後の予定・確度など）はそのnoteをhintとして使用
- **備考**のみコード側で個別指定：「原文のまま・省略せずにすべて入れること。要約・省略禁止」。さらに【重要】セクションにも念押しで同内容を追記

### 6-2. Groq②：複数案件判定（ヒソヒソくん用）
Yes/Noのみ返す。トークン節約・判定ブレ防止。

```json
{"is_multiple": true または false}
```

判定基準：
- 会社名が複数登場する → true
- 「あと〜」「別件で〜」などの切り替えワードがある → true
- 上記に当てはまらない → false
- 迷ったら false（止めない・通す方針）

### 6-3. Groq③：プルプル部長
キャラクター設定をsystem_promptに埋め込む。時間・曜日コンテキストはPython側で判定してプロンプトに追加。

| 条件 | 部長の動作 |
|------|-----------|
| 平日18時未満 | 通常モード（武勇伝・ゴルフ・バブル時代・ケチ一言＋承認） |
| 平日18時以降 | コンプライアンス一言 → 昔話に脱線 |
| 休日 | コンプライアンス一言 → 昔話に脱線 |

出力形式：
```json
{"stamp": "承認：プルプル部長", "comment": "部長の一言"}
```


## 7. ヒソヒソくんの動作設計

| 判定方法 | 条件・発言 |
|---------|-----------|
| Python側 | 打合せ日が空 → 今日の日付を自動補完して「今日にしときました」と報告 |
| Python側 | 他の必須項目が空 →「…✱の{項目名}、聞けましたか？（小声）」を全件列挙 |
| Groq②側 | 複数案件混在 →「たくさんご要望がありますね。。（小声）」 |
| どちらでもない | 何も言わない（沈黙） |

### 設計思想
- **止めない・通す方針**。気づかせるだけでユーザーに判断を委ねる
- **必須項目の空欄は複数あっても全部指摘する**（Salesforceユーザーは「一度で全部把握したい」ため）
- コピーも部長承認も、必須欠けのまま実行できる
- 任意項目が空でもヒソヒソくんは何も言わない（メリハリをつける）
- メッセージは1つの枠内に複数行まとめて表示（`<br>` 結合）
- `st.empty()` で同じ場所を上書き表示（ズラズラ増えない）


## 8. キャラクター設定

### 8-1. プルプル部長 🐼
- 58歳、昭和の営業マン気質
- 口癖：「わしの時代はな…」
- メモに人名があれば必ず反応する（知ってるかどうかは関係ない）
- ケチをつけるが最終的には必ず承認する
- ケチのネタ：ひらがな・句読点・文字数・改行・カタカナ英語
- 雑談ネタ：ゴルフ・昔の接待文化・バブル時代・最近の若者への愚痴
- 18時以降・休日はコンプライアンスに配慮（ギャップ演出）

### 8-2. ヒソヒソくん 🐰
- ヒソヒソ話すが内容は的確。実は一番仕事できる
- 話し方：「…あの、ちょっとよろしいですか（小声）」
- 入力する人の味方。指摘はするがうるさくしない


## 9. 項目定義JSON（商談用・確定版）

```json
{
  "template_name": "商談レコード",
  "fields": [
    { "name": "打合せ日",       "required": true },
    { "name": "打合せ時間",     "required": false },
    { "name": "自社参加者",     "required": false },
    { "name": "会社名",        "required": true },
    { "name": "担当者名",       "required": true },
    { "name": "商談フェーズ",   "required": true,
      "options": ["初回接触", "提案中", "見積提出", "クローズ"] },
    { "name": "予算感",         "required": false },
    { "name": "導入希望時期",   "required": false,
      "note": "四半期・年度など大きな粒度の時期のみ。例：1Q、上期、来年度。具体的な月日はここに入れない" },
    { "name": "今後の予定",     "required": false,
      "note": "具体的な月・日付・マイルストーン。例：4月キックオフ、5月末、来月中。四半期表記はここに入れない" },
    { "name": "確度",           "required": false,
      "options": ["高", "中", "低"],
      "note": "メモに明記されている場合のみ入力。不明な場合は必ず空文字" },
    { "name": "競合情報",       "required": false },
    { "name": "次のアクション", "required": true },
    { "name": "備考",           "required": false }
  ]
}
```

各フィールドの設計意図：
- **打合せ日**：必須。空欄時はヒソヒソくんが今日の日付を自動補完
- **導入希望時期**：四半期・年度など大きな粒度のみ（1Q、上期、来年度など）
- **今後の予定**：具体的な月・日付・マイルストーン（月日が書かれたスケジュール）
- **備考**：どの項目にも分類できなかった情報の受け皿。省略・要約禁止で原文のまま入れる。複数案件が混ざった場合も一目でわかる


## 10. 用語辞書JSON（glossary.json）

```json
{
  "glossary": [
    { "term": "1Q", "description": "第1四半期（4月〜6月）", "field": "導入希望時期" },
    { "term": "2Q", "description": "第2四半期（7月〜9月）", "field": "導入希望時期" },
    { "term": "3Q", "description": "第3四半期（10月〜12月）", "field": "導入希望時期" },
    { "term": "4Q", "description": "第4四半期（1月〜3月）", "field": "導入希望時期" }
  ]
}
```

- 仕組みだけ最初から作り、使いながら追記していく運用
- 例追加：`{ "term": "MM単価", "description": "月単価のこと", "field": "予算感" }`


## 11. 実装済みUI詳細

### テーマカラー
- ブルーグレー（#455a64 / #546e7a / #78909c）× たまご色・オレンジの組み合わせ
- 背景：doodle風JPG画像（assets/）をbase64エンコードしてCSS注入

### タイトルバー
- `.stApp::before` 疑似要素で実装（高さ30px・固定・全幅）
- **注意**：`backdrop-filter` を持つ親要素が containing block になり `position:fixed` が効かなくなる問題があるため、`div` ではなく `::before` 疑似要素を使う

### ガラスパネル
- セレクタ：`[data-testid="stMainBlockContainer"]`
- `background: rgba(255,255,255,0.22)` + `backdrop-filter: blur(10px)`

### ルーズリーフリング
- テキストエリアの直上に9個のリングをflex均等配置
- `margin-bottom: -6px` でリングとエリアを密着させる
- CSSは `radial-gradient` + `box-shadow` でメタル質感を表現

### テキストエリア（付箋スタイル）
- 背景：#FFFDE7、ボーダー：#FFD54F
- `caret-color: #333` と `::placeholder { color: #bbb; opacity: 1 }` を明示（iOS Safari ダークモード対策）
- **`key="memo_text"` を使用**（`value=` は使わない）。`value=` を毎回渡すと再レンダリング時にコピペ内容が上書き消滅する問題があるため

### 整形結果パネル（バインダー風）
- `::before`：クリップのワイヤー部分（border: 3px solid #757575）
- `::after`：クリップ本体（linear-gradient グレー系）
- 値の改行表示：HTMLエスケープ後に `.replace("\n", "<br>")` を適用（備考の複数行が正しく表示される）

### ハンコ（部長ページ）
- 円形・朱色・かすれ風（opacity + filter: blur(0.4px)）
- `math.radians()` + `sin/cos` で🌸を8方向（45°間隔）に配置
- アニメーション：scale 1.3 → 0.95 → 1.0 で着地

### コピーボタン
- `st.components.v1.html()` を使用（`st.markdown` ではJSが無効化される）
- iframeに `<meta name="color-scheme" content="light">` を付与（iOS Safari ダークモードでボタンが消える問題の対策）

### ボタンスタイル
- primary（整形ボタン）：オレンジ #e8802a
- secondary 有効（メインページ：部長ボタン）：黄色 #FFD600
- secondary 有効（部長ページ：戻るボタン）：白 background + #546e7a テキスト（ダークモード対策で background-color: white を明示）
